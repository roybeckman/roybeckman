Artificial Hierarchy Memory (AHM) – Specification v1.0
1. Introduction & Vision
Artificial Hierarchy Memory (AHM) is a self-describing, serverless, and portable project memory format designed for seamless AI-to-AI project transfer (pp. 1-2).
Traditional AI memory solutions rely heavily on heavy infrastructure, such as external vector databases (RAG), central servers, or platform-specific APIs. AHM bypasses these requirements by shifting the complexity from the software infrastructure to the structural design of a single text file (p. 3).
Core Principles
    • The Instruction Travels with the Memory: Every AHM file contains both the system execution prompt and the relational dataset within the exact same payload (pp. 2, 4).
    • Project Continuity, Not Restart: A project should not restart when the AI model or agent changes. A project should continue (pp. 1, 4).
    • Serverless Portability: An AI system requires zero external databases, plugins, or software to read, decode, and execute an AHM file (p. 3).
    • Semantic DNA: Instead of parsing natural language narrative text, incoming AI agents read a pre-compressed, token-efficient network of logical relationships (p. 2).

2. Structural Architecture
Every compatible AHM file must strictly follow a 5-tier chronological layout (p. 4). An AI system must be able to understand the current project state dynamically upon reading the file for the very first time (pp. 1, 4).

2.1 INSTRUCTIONS
This section must always reside at the absolute beginning of the file (p. 4). It commands the consuming LLM agent to initialize its system instructions, decompress the dataset using the local dictionary key, and append updates using the identical syntax without destroying the structure (p. 1).
2.2 DESCRIPTION
Defines the technical scope, project identification title, current evolutionary phase, and the ultimate milestone objectives (p. 1).
2.3 KEY
The standardized translation dictionary (pp. 1, 4). It maps single characters or short strings to abstract semantic archetypes to allow for text compression (pp. 1-2).
    • P = Project (p. 1)
    • O = Object (p. 1)
    • F = Fact (p. 1)
    • R = Relation (p. 2)
    • S = State (p. 2)
    • N = Next (p. 2)
    • K = Knowledge (p. 2)
    • A = Action (p. 2)
    • IB = Information Bank (p. 2)
2.4 MEMORY (RB_COMPRESS)
The core working memory layer (pp. 2, 4). It uses Dictionary-Based Semantic Compression (RB_COMPRESS) to compress pure meaning rather than string text (pp. 2, 4).
Relationships are mapped out via direct mathematical links (R:ENTITY_A>ENTITY_B:RELATIONSHIP) (p. 3).
Mathematical Extensions (?, ?, STAT)
To measure agent adaptation over time without conversational histories, this layer supports dynamic state evaluation:
    • STAT:[NAME]:[VALUE]: Numerical values reflecting continuous state metrics (e.g., STAT:COMPLETION:0.45).
    • $\gamma$ (Gamma): Learning rate, priority weighting, or importance decay factor.
    • $\delta$ (Delta): Metric shifts indicating recent momentum change or user status updates.
2.5 INFORMATION_BANK
A raw, uncompressed background knowledge repository used exclusively for full project context recovery if semantic anomalies or ambiguities occur within the logical memory layer (pp. 3-4). This section must always exist, even if it contains no records (p. 4).

3. Protocol Execution Flow
When an AI agent initializes an AHM file, it must process the nodes sequentially:
    1. Parse Instructions: Read the processing sequence rules (p. 1).
    2. Load Key: Cache the token compression mapping dictionary (p. 1).
    3. Expand & Ingest: Unpack the MEMORY network to map out project relations immediately (pp. 1, 3).
    4. Evaluate State: Read current STATE and target NEXT actions (p. 1).
    5. Process Context: Consult the INFORMATION_BANK if entity background resolution is needed (p. 1).
    6. Execute & Append: Work on the target tasks, recalculate delta adjustments, and save file updates back into the identical AHM syntax (p. 1).
